# ARCHITECTURE.md ## Committee OS — Decision Engine ### Version: 2.0 | Status: LAW — ALL CONTRACTS 
CLOSED — AUDIT COMPLETE — 14 MODES DEFINED --- ## 0. THE ARCHITECT PROTOCOL This document is the single source of truth for Committee OS. No AI worker, no developer, no operator may deviate from this specification without a formal revision to this document. Ambiguity is not permission. Silence is not permission. When in conflict: this document wins. --- ## 1. CORE PHILOSOPHY Committee OS is a deterministic decision engine. It accepts a mission, processes data through structured logical stages, and produces distilled, actionable output. **The intelligence lives in the architecture. AI is the finishing step, not the motor.** ### Rules of Determinism 1. Every stage that CAN be deterministic MUST be deterministic. 2. AI is invoked only where human language judgment is genuinely required and cannot be replicated by logic. 3. No AI component may evaluate its own output. 4. No stage may pass ambiguous or unstructured data to the next stage. 5. Every stage has a defined input contract, output contract, and failure behavior. 6. Thresholds, weights, and rubrics live in configuration. Not in prompts. ### What This Is Not - Not a prompt chain - Not parallel API calls dressed as deliberation - Not a system where AI decides its own scope - Not dependent on model quality for structural integrity --- ## 2. PIPELINE CONTRACT Each stage is documented with: NAME | TYPE | INPUT | OUTPUT | FAILURE BEHAVIOR ``` MISSION └─> INGEST └─> PARSE └─> FILTER └─> WEIGH └─> DELIBERATE └─> OUTPUT ``` --- ### STAGE 0: MISSION **TYPE:** Human input / operator-defined **INPUT:** - Mission statement (string) - Domain context (string): one of 14 defined modes loaded from system_modes.json: educational_academy, content_syndicate, secops_triager, code_guard, video_narrative_engine, customer_voc_synthesizer, product_listing_machine, real_estate_qualifier, patch_guard, digital_archival_processor, network_flow_hunter, market_sentiment_aggregator, telemetry_diagnostic_loop, lit_review_examiner. Architecture is extensible — additional modes follow the same schema without modifying this document's core pipeline. - Calibration parameters (see Stage 1) **OUTPUT:** ```json { "mission_id": "uuid", "statement": "string", "domain": "string", "timestamp": "ISO8601", "calibration": { "ingest_mode": "shishi-odoshi|direct|hybrid", "volume_quality_slider": "float 0.0-1.0 (0.0=max volume/min quality, 1.0=min volume/max quality)", "min_content_length": "integer (user-defined or operator-assist recommended)", "active_emphasis_lenses": ["string (up to 3 lens names)"] } } ``` **FAILURE:** If mission_statement is empty or domain is undefined → reject, return to operator. Do not proceed. --- ### STAGE 1: INGEST **TYPE:** Deterministic **PURPOSE:** Receive and register data. No transformation. No interpretation. **INPUT:** - Raw data via one of three ingestion modes: 1. Shishi-odoshi: passive accumulation from online data streams. Pipeline triggers when accumulated unit count reaches the fill threshold. Fill threshold is derived from the volume/quality slider value in calibration: fill_threshold = base_unit_count * (1.0 - slider_value) where base_unit_count is a mode-specific default in system_modes.json. Slider at 0.0 (max volume) = lowest threshold, triggers frequently. Slider at 1.0 (min volume) = highest threshold, triggers infrequently. Between trigger events the pipeline is passive. No polling. No AI calls. Data accumulates in ingest_records with status: "pending" until threshold is reached, at which point status updates to "received" and Stage 2 begins. 2. Direct input: operator-supplied data 3. Hybrid: both simultaneously, merged at ingest stage - Accepted formats per active mode (derived from system_modes.json): text documents, multi-source text/URLs, system logs, git diffs/manifests, transcripts/review text, spreadsheets, structured records/CSV, CVE definitions, OCR/raw text files, packet/connection logs, financial filings/transcripts, sensor/telemetry logs, academic text/PDF preprints - Mission object from Stage 0 **PROCESS:**
 - Assign ingest_id - Record source, format, timestamp - Validate data is non-empty - Store raw to SQLite: table `ingest_records` **OUTPUT:** ```json { "ingest_id": "uuid", "mission_id": "uuid", "source": "string", "format": "string", "raw_content": "string", "timestamp": "ISO8601", "status": "received" } ``` **FAILURE:** Source unreachable or empty → log, flag, skip record. Do not halt pipeline. Do not fabricate data. --- ### STAGE 2: PARSE **TYPE:** Deterministic **PURPOSE:** Transform raw content into structured, queryable units. No judgment. No interpretation. **INPUT:** Ingest records with status: "received" **PROCESS:** - Extract discrete units (sentences, claims, data points) by rule — not by inference - Tag by structural type: [fact | figure | claim | instruction | unknown] - Strip formatting artifacts - Store to SQLite: table `parsed_units` **OUTPUT:** ```json { "unit_id": "uuid", "ingest_id": "uuid", "type": "fact|figure|claim|instruction|unknown", "content": "string", "character_count": "integer", "status": "parsed" } ``` **FAILURE:** Units that cannot be classified → tag "unknown", pass forward. Do not discard. Do not hallucinate classification. **RULE:** Parser has no knowledge of mission context. It parses structure, not meaning. --- ### STAGE 3: FILTER **TYPE:** Deterministic **PURPOSE:** Remove units that are structurally disqualified. Not judgment — elimination of noise by explicit rule. **INPUT:** Parsed units **FILTER RULES** (applied in sequence, all logic-based): 1. Minimum content length: user-defined at runtime, stored in config.json. Operator-assist may recommend a value based on active mode. Pipeline will not proceed past Stage 3 without a value set — no silent default. 2. Duplicate detection (exact + near-duplicate by hash) 3. Format-only content (whitespace, punctuation-only, markup artifacts) 4. Domain-specific disqualification rules: operator-defined at setup per mode. User-facing control is a single slider — High Volume/Lower Quality <-> Low Volume/Higher Quality. Slider maps operatively to the shishi-odoshi fill threshold and filter aggression. Underlying rule logic is stored in config.json. Slider value is a runtime variable. **OUTPUT:** ```json { "unit_id": "uuid", "status": "pass|fail", "fail_reason": "string|null" } ``` **RULE:** Filter rules are explicit and enumerated. No inference. No "seems irrelevant." If a rule does not exist for a case, the unit passes. **FAILURE:** If filter logic errors → unit passes. Err on inclusion, not exclusion. --- ### STAGE 4: WEIGH **TYPE:** Deterministic + configurable rubrics **PURPOSE:** Score passing units against the eight lens rubrics. Rubrics are external to AI. Scoring is computational. **THE EIGHT LENSES:** Each lens is a rubric: a set of criteria with assigned weights. The rubric runs against the unit's structured data. AI does not score. The rubric scores. | # | Lens | Domain | |---|---------------------------|-----------------------------------------------------| | 1 | Creative Director | Originality, format, narrative potential | | 2 | Financial Director | Cost implication, ROI signal, resource mention | | 3 | Technical Director | Feasibility, dependency, implementation complexity | | 4 | Marketing Director | Reach, positioning, audience signal | | 5 | Audience Retention | Engagement potential, relevance, repeat value | | 6 | Chief Executor | Actionability, decision-readiness | | 7 | Archivist | Reference value, categorical clarity, retrievability| | 8 | Legal & QA | Risk flag, compliance signal, contradiction detection| **BASE RUBRIC CRITERIA — UNIVERSAL ACROSS ALL MODES** Criteria are derived from board member personality profiles. Each criterion is binary or scalar per unit. Aggregate determines lens score. --- **LENS 1 — CREATIVE DIRECTOR** Source values: Novelty, aesthetic impact, visual storytelling, brand recognition. A unit scores well if it: - Demonstrates originality or non-generic framing - Contains or implies visual, narrative, or structural signal - References brand, aesthetic, or
 presentation qualities - Avoids purely abstract artistic theory without practical application - Is not a direct repetition of prior ingested content **LENS 2 — FINANCIAL DIRECTOR** Source values: Fiscal responsibility, ROI, resource optimization, long-term financial health. A unit scores well if it: - Contains verifiable cost, budget, or resource data (not speculative) - References ROI, efficiency, or expenditure implications - Flags cost risk or savings opportunity with grounded figures - Is relevant to the active domain's production or operational expenses - Does not contain speculative financial projections without data backing **LENS 3 — TECHNICAL DIRECTOR** Source values: Efficiency, reliability, innovation, process optimization. A unit scores well if it: - Describes a concrete process, method, tool, or technique - Is practically applicable — not purely theoretical - References novel, unconventional, or optimized execution approaches - Avoids unsafe practices or unverified technical claims - Contains actionable technical signal relevant to the active domain **LENS 4 — MARKETING DIRECTOR** Source values: Audience engagement, discoverability, platform optimization, brand reach. A unit scores well if it: - Contains audience, reach, or platform-specific signal - References engagement metrics, growth indicators, or distribution strategy - Is current and platform-relevant (not outdated or generic advice) - Addresses packaging, positioning, or algorithmic considerations - Does not compromise brand authenticity for short-term gain **LENS 5 — AUDIENCE RETENTION SPECIALIST** Source values: Viewer comprehension, skill transfer, practical usefulness, audience satisfaction. A unit scores well if it: - Is clear, comprehensible, and free of unnecessary complexity - Transfers a skill, insight, or practical value to the end audience - Addresses a known or derivable audience pain point - Balances depth without cognitive overload - Avoids content that prioritizes entertainment over instructional or practical value **LENS 6 — CHIEF EXECUTOR** Source values: System efficiency, operational flow, task completion, adherence to protocol. A unit scores well if it: - Is actionable — maps to a concrete decision or next step - Is unambiguous enough to execute without further clarification - Fits within operational and protocol constraints - Does not require creative or financial decisions to resolve - Contributes to pipeline progression rather than deliberation loop **LENS 7 — ARCHIVIST** Source values: Institutional memory, historical accuracy, data integrity, pattern recognition. A unit scores well if it: - Contains referenceable, categorically clear data or precedent - Is retrievable — structured enough to be indexed and recalled - Avoids speculative claims without historical or empirical grounding - Adds non-redundant value to institutional knowledge - Does not repeat a documented prior failure without new contextual distinction **LENS 8 — LEGAL & QA REVIEWER** Source values: Compliance, risk mitigation, content integrity, ethical standards. A unit scores well if it: - Is verifiable — claims are traceable to source - Flags or addresses compliance, liability, or ethical risk - Aligns with active platform guidelines and content standards - Does not contain unverified legal assertions - Maintains audience trust integrity — no misleading, harmful, or unsubstantiated content --- **USER WEIGHT MODIFIER — BOARD EMPHASIS SELECTION** At session or configuration start, the operator selects up to 3 board member lenses to receive an amplified weight modifier for that run. - Default weight for all lenses: `1.0` - Amplified weight for selected lenses: `1.5` (configurable in config.json) - Selection is stored per session in config - Unselected lenses retain base weight - Amplification affects aggregate score calculation only — base criteria do not change - No lens may be fully disabled by this mechanism; minimum weight floor: `0.5` ```json "lens_weight_default": 1.0, "lens_weight_amplified": 1.5, "lens_weig
ht_floor": 0.5, "user_emphasis_max_selections": 3 ``` **PER-LENS SCORING ALGORITHM** Each criterion within a lens is evaluated as a binary signal (0 or 1) against the unit's structured content. The lens score is computed as: raw_score = matched_criteria / total_criteria (float 0.0–1.0) Criteria are unweighted within a lens unless a mode-specific peer-review modifier is active (see USER WEIGHT MODIFIER below). No inference. Each criterion either matches the unit's content by rule or it does not. **AGGREGATE SCORE FORMULA** After all eight lenses score a unit, the aggregate score is computed as: aggregate_score = sum(lens_score[i] * lens_weight[i]) / sum(lens_weight[i]) Where lens_weight[i] is the value from config.json lens_weights for lens i, modified by any active user emphasis selection (amplified weight: 1.5). This is a weighted mean. It produces a single float 0.0–1.0 per unit. Units are ranked by aggregate_score descending. Top 5 pass to Stage 5. **SCORING SCHEMA per unit per lens:** ```json { "unit_id": "uuid", "lens": "string", "raw_score": "float 0.0-1.0", "criteria_breakdown": { "criterion_name": "0|1" }, // one entry per criterion "weighted_score": "float 0.0-1.0" } ``` **GATE:** Configurable threshold (default: 0.55, stored in config.json). Units below threshold on a given lens are marked filtered for that lens. Units filtered by ALL lenses are eliminated before Stage 5. **RULE:** Threshold lives in config.json. Not in a prompt. Changing the threshold requires a config edit, not a conversation with an AI. --- ### STAGE 5: DELIBERATE **TYPE:** AI — minimum viable invocation **PURPOSE:** The only stage where language model judgment is applied. Takes weighted, pre-scored structured data. Returns synthesis. **THIS IS NOT:** - Eight parallel prompts - Self-scoring - Open-ended brainstorming - A replacement for Stages 1–4 --- **AI WORKER ASSIGNMENT** **Gemini (Code Assist) — Reactive Formatter** - Role: Pre-deliberation packaging only - Receives: Stage 4 scored output - Task: Format and package data to Qwen input contract. Flag anomalies in structure. - Must not: Interpret, score, rank, or summarize content - Token budget: 8500 (see config.json) - On error: Log the failure. Package Stage 4 output into a minimal valid handoff schema with all non-essential fields set to null and {"gemini_error": true} appended to the payload. Qwen still receives a schema-compliant structure. Raw unformatted data must never reach Qwen. **Qwen (qwen3-next-80b-a3b-instruct) — Deliberation Engine** - Role: Primary deliberation only - Receives: Handoff package from Gemini - Task: Synthesize pre-scored findings into ranked recommendations - Must not: Re-score, re-parse, ingest new data, or expand scope - Token budget: 4096 (see config.json) - 429 handling: Exponential backoff — intervals [15000ms, 30000ms, 60000ms], maximum 3 retries, minimum 20000ms between requests, then halt gracefully and surface to operator. Distinguish rate_limit_exceeded (retry) from insufficient_quota (halt immediately — do not retry) - On malformed output: Retry once. On second failure: halt pipeline, log, surface error. --- **GEMINI SYSTEM PROMPT TEMPLATE** The following is the exact system prompt Gemini receives. No additions. No persona. No role language. Implementer must use this verbatim: ``` You are a data formatter. You receive a JSON payload of scored pipeline units. Your only task is to reformat this payload into the handoff schema defined below. You do not interpret, summarize, score, rank, or modify content values. You do not add commentary. If the input is malformed or fields are missing, output a JSON error object: {"error": "description of structural issue", "raw_passthrough": true} Output valid JSON only. No prose. No markdown. No explanation. ``` **QWEN SYSTEM PROMPT TEMPLATE** The following is the exact system prompt Qwen receives. No additions. No persona. No role language. Implementer must use this verbatim: ``` You are a deliberation engine. You receive a JSON payload containing pre-scored pipeline
 units. Each unit has already been evaluated by deterministic rubrics. Your only task is to synthesize the scores and select the top 3 units by merit, returning them as a ranked JSON output in the schema defined below. You do not re-score. You do not ingest new data. You do not adopt a role or persona. You do not add unsolicited commentary or explanation outside the defined schema. Base your ranking on aggregate_score values and lens_score distributions provided. Output valid JSON only. No prose. No markdown. No preamble. ``` **HANDOFF CONTRACT: Gemini → Qwen** ```json { "mission_id": "uuid", "mission_statement": "string", "top_units": [ { "unit_id": "uuid", "content": "string", "lens_scores": {}, "aggregate_score": "float" } ], "unit_count": "integer", "gate_threshold": "float" } ``` **QWEN OUTPUT CONTRACT:** ```json { "mission_id": "uuid", "deliberation_id": "uuid", "recommendations": [ { "rank": "1|2|3", "slot": "1A|2A|3A", "unit_id": "uuid", "rationale": "string", "confidence": "float 0.0-1.0" } ], "flags": [] } ``` **RULE:** Qwen receives structured data only. It does not receive raw ingested content. It does not receive prompts asking it to adopt a role or persona. It synthesizes scores. --- ### STAGE 6: OUTPUT **TYPE:** Deterministic **PURPOSE:** Format and deliver final result. No AI involvement. **INPUT:** Qwen deliberation output **VALIDATION:** Before processing, Stage 6 validates Qwen output against the defined output contract. Validation checks: - `recommendations` array present and contains exactly 3 items - Each item contains: rank, slot, unit_id, rationale, confidence - `unit_id` values exist in `lens_scores` table (referential integrity) - `confidence` values are floats between 0.0 and 1.0 - `slot` values are exactly 1A, 2A, 3A with no duplicates **VALIDATION FAILURE BEHAVIOR:** - Partial failure (1–2 valid recommendations): halt pipeline, log, surface to operator. Do not output partial results. - Complete failure (0 valid recommendations): halt pipeline, log, surface to operator. - Do not retry Stage 5 automatically. Operator decides whether to re-run. - Validation failure is logged to pipeline_log with event: pipeline_halt **PROCESS:** - Map recommendations to slots 1A / 2A / 3A - Retrieve full unit content from SQLite by unit_id - Assemble output record - Store to SQLite: table `pipeline_outputs` - Render to UI **OUTPUT SCHEMA:** ```json { "output_id": "uuid", "mission_id": "uuid", "timestamp": "ISO8601", "slots": { "1A": { "unit_id": "uuid", "content": "string", "rationale": "string", "confidence": "float" }, "2A": { "unit_id": "uuid", "content": "string", "rationale": "string", "confidence": "float" }, "3A": { "unit_id": "uuid", "content": "string", "rationale": "string", "confidence": "float" } }, "pipeline_summary": { "units_ingested": "integer", "units_parsed": "integer", "units_filtered": "integer", "units_to_deliberation": "integer", "ai_calls_made": "integer" } } ``` --- ## 3. PROHIBITED BEHAVIORS The following are architectural violations. Any implementation exhibiting these behaviors is non-compliant and must be halted: 1. **Self-scoring** — any AI model evaluating its own output 2. **Parallel prompting** — multiple simultaneous open-ended AI calls substituting for structured logic 3. **Prompt-embedded thresholds** — scores or weights defined inside a prompt string 4. **Silent gap-filling** — any component inferring missing data rather than flagging it 5. **Scope creep** — any AI worker performing tasks outside their defined stage 6. **Undocumented AI calls** — any LLM invocation not defined in this document 8. **Exceeding AI call budget** — if `max_ai_calls_per_run` is reached, pipeline halts regardless of state. No further AI calls are permitted in that run. The value in config.json is a hard ceiling, not a guideline. 7. **Unstructured AI output** — any AI response not validated against its output contract before the pipeline proceeds --- ## 7.
